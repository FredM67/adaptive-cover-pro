# v2.23.3

Restores correct cover dispatch when a user-configured floor clamps the pipeline output and interpolation or inverse-state is active.

---

## 🐛 Fixes

### Floor-clamped position now bypasses interpolation and inverse-state remapping ([#469], [#471])

When a cover had `sunset_position=0`, a min-mode custom position of 25%, and a non-linear calibration curve (`interp_list_new=[0,26,58,100]`), the pipeline correctly reported `position=25` in `pipeline_evaluated` — but `cover_command_sent` dispatched 45%. On every subsequent tick, `cover_command_skipped` fired because the current position was already 45, while the pipeline continued to say 25. The cover was stuck above its floor.

The root cause: after the registry's floor-composition pass raises a winner's position to the user-configured floor, that value is already in cover-position space. The coordinator's `state` property then ran it through interpolation and inverse-state remapping as if it were a raw calculation output, re-mapping the 25% floor to 45% on the calibration curve before dispatch. The same defect affected inverted covers — a floor of 25% on an inverted cover would produce a dispatch of 75%.

The fix adds a `floor_clamp_applied: bool` field (default `False`) to `PipelineResult`. `PipelineRegistry` sets it to `True` during floor composition whenever the winner's position is raised to the floor. In `AdaptiveDataUpdateCoordinator`, the `state` property's post-pipeline short-circuit now gates on `floor_clamp_applied` alongside the existing `bypass_auto_control` path, returning the position directly without further transformation. The flag is also written into the `decision_trace` under the `floor_composition` step so it is visible in diagnostics.

This was a regression introduced in v2.23.0 by #465 (the pipeline-floor refactor). Users running v2.23.0–v2.23.2 with a min-mode custom position **and** interpolation **or** inverse-state enabled should upgrade.
