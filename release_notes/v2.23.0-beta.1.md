# v2.23.0-beta.1

v2.23.0 is the first beta of a new minor release line. The headline feature is `sensor.<cover>_position_forecast` ŌĆö a diagnostic sensor that projects today's solar-tracking behavior as a coarse timeline the companion Lovelace card can render as a position strip. Alongside that, this release adds HA unit-system support across the config UI and sensors, fixes a My Position routing bug that silently dropped button presses when automatic control was off, and closes five cooperating defects that had kept the forecast sensor frozen after boot. Feedback welcome before this promotes to stable.

## ­¤Ä» Highlights

- **Position forecast sensor** ŌĆö `sensor.<cover>_position_forecast` shows the next forecast event as its state and carries a `forecast` attribute (15-min samples over a 12-hour window) plus an `events` list (sunrise, sunset, FOV enter/exit). Built purely from solar geometry, so it degrades gracefully when override inputs aren't known in advance.
- **Frozen forecast fixed** ŌĆö five bugs cooperated to keep the forecast sensor at its boot-time value on live installations: tasks garbage-collected before running, the tick function dispatched to a worker thread, ticks unaligned to wall-clock boundaries, listeners never notified after recompute, and FOV events lagging the position curve. All five are fixed.
- **HA unit system** ŌĆö geometry fields in the config UI and `sensor.<cover>_target_position` distance attributes now respect the user's HA unit system. US-customary instances see inches; storage stays in metres so no migration runs.
- **My Position button now works with automatic control off** ŌĆö pressing the button when the automatic-control switch is off no longer silently drops the command.
- **Companion card slot management** ŌĆö per-slot enable/disable, a `custom_position_slots` snapshot on the decision trace, slot sensor friendly-name, and the position-forecast sensor all support [adaptive-cover-pro-card#60](https://github.com/jrhubott/adaptive-cover-pro-card/issues/60).
- **My-preset entities gated by config-flow toggle** ŌĆö the button and number for My Position are now off by default and gated behind a config-flow toggle, reducing clutter for installs that don't use Somfy My.
- **Performance** ŌĆö two improvements reduce per-forecast cost for multi-cover instances: an O(1) arithmetic index lookup replaces a linear scan, and `SunData` is shared across all entries at the same location.

## Ō£© Added

### Position forecast sensor ([#432], PR [#432])

A new `sensor.<cover>_position_forecast` diagnostic sensor (device class `timestamp`) projects today's solar-tracking behavior as a coarse timeline. Its state is the ISO 8601 timestamp of the next upcoming forecast event. Its `forecast` attribute carries a list of `{t, position, handler}` samples at 15-minute steps over a 12-hour window; its `events` attribute lists `{t, kind, label}` boundary events where `kind` is one of `"sunrise"`, `"sunset"`, `"fov_enter"`, `"fov_exit"`.

The `forecast.py` module exposes `build_forecast()` as a pure function parameterized over a `cover_factory` closure (solar azimuth + elevation ŌåÆ `AdaptiveGeneralCover`), and `build_forecast_for_coord()` as a thin shim that wires it to the coordinator's sun provider, policy, and config service. Only solar tracking is projected ŌĆö manual override, motion, weather safety, and custom positions depend on real-time inputs that would mislead a forecast if naively held at their current state. The sensor degrades gracefully to `None` on any setup-time exception via `_safe_forecast`, so it stays available during the first-refresh window. Translations added to `en.json`, `de.json`, and `fr.json`.

### Custom-position slot enable/disable + decision-trace slot snapshot ([#432])

Each custom-position slot gains a new opt-out config key `custom_position_enabled_<N>` (default `True` when absent, so existing entries behave unchanged). Setting it `False` silences a slot without losing its sensor, position, or priority configuration. Both `PipelineSnapshotBuilder.read_custom_position_sensors` and `_build_pipeline()` in the coordinator honor the flag; a disabled slot is excluded from the snapshot and never instantiated as a `CustomPositionHandler`.

The `set_custom_position` service gains an `enabled` field that writes `custom_position_enabled_<N>` so card and automation callers can toggle a slot without opening the options flow.

`sensor.<cover>_decision_trace` gains a new attribute `custom_position_slots` ŌĆö a stable 4-row list (one per slot, configured or not) with the shape `{slot, enabled, sensor, sensor_name, position, priority, min_mode}`. Unconfigured slots appear with `enabled=False` and other fields null so the card can render a fixed grid.

`sensor.<cover>_decision_trace` also gains `custom_position_active_slot_name` when a `CustomPositionHandler` wins the pipeline. The value is the winning slot's bound binary sensor's `friendly_name` attribute, pulled via `ATTR_FRIENDLY_NAME` from `hass.states`. Falls back to `None` when the sensor has no friendly name; the attribute is omitted entirely in that case to keep the trace clean. The field is threaded through `CustomPositionSensorState.sensor_name`, `PipelineResult.custom_position_active_slot_name`, `CustomPositionHandler`, and `_decision_trace_attrs` in `sensor.py`.

### `active_slot` and `floor_binding` on the decision trace ([#421], PR [#424])

`sensor.<cover>_decision_trace` gains two new attributes when a Custom Position handler wins the pipeline:

**`custom_position_active_slot`** ŌĆö the 1-based slot number (1ŌĆō4) of the winning Custom Position handler. `None` when any other handler wins.

**`custom_position_minimum_mode`** ŌĆö present only when `custom_position_active_slot` is set and `min_mode` is active. `True` when the configured floor is raising the cover above the raw autonomous calculation (the floor is actively constraining). `False` when the raw calculation already meets or exceeds the floor (the floor is a no-op). `None` in exact mode (`min_mode=False`) or when any non-custom handler wins.

`PipelineRegistry.evaluate` uses `dataclasses.replace` to build the `PipelineResult` from the winning handler, so all fields on the winner carry through without being enumerated individually.

### HA unit system in config UI and sensors ([#428])

Geometry and sun-tracking configuration now displays in the user's HA unit system. When HA is configured as US-customary, length fields (blind height, awning length, slat dimensions, shaded distance) show in inches rather than decimal metres. The `weather_override_schema` and related weather-step schemas now label wind-speed and rain thresholds with the configured sensor's `unit_of_measurement` when available, falling back to HA's locale unit.

Storage remains canonical (metres, centimetres, sensor-unit) ŌĆö no migration needed for existing entries.

`sensor.<cover>_target_position` gains three new attributes: `target_distance`, `actual_distances`, and `distance_unit`, translating the 0ŌĆō100% position into a physical travel distance using the cover's lift-axis dimension (window height for blinds/venetian, awning length for awnings). Tilt-only covers skip these attributes. The cover-type dispatch lives behind a new `lift_travel_metres` hook on `CoverTypePolicy` so `sensor.py` stays free of cover-type branching.

`sensor.<cover>_climate_status` gains a `ha_temperature_unit` attribute alongside the existing `temperature_unit`.

### My-preset entities gated by config-flow toggle ([#434])

The My Position button and My Position Value number entities are now off by default and gated behind a config-flow toggle. Existing entries where the entities were already configured keep them visible; new entries don't surface the button/number until the toggle is enabled. Reduces clutter for installs that don't use Somfy My or a hardware "My" preset concept.

### Shaded distance 0 m and glare zone Z height ([#429])

**Distance 0 m:** `_RANGE_DISTANCE` is lowered from `(0.1, 50.0)` to `(0.0, 50.0)` and the `length_selector` floor matches. The calculation engine multiplies by distance and never divides, so distance = 0 yields position = 0 (fully closed at the glass plane) with no NaN or inf.

**Glare zone Z height:** `GlareZone` gains a `z: float = 0.0` field. When `z > 0`, `glare_zone_effective_distance` in `engine/covers/vertical.py` offsets the effective distance by `z / tan(sol_elev)` using the shared `_elevation_offset` helper ŌĆö the same trigonometric construction as the sill-height offset ŌĆö so you can protect a point above the floor (eye level, a TV screen, a tabletop) rather than just a floor disk. Default `z = 0` reproduces existing behaviour byte-for-byte. The `glare_zone_<N>_z` key is added to `OPTION_RANGES` and `FIELD_VALIDATORS`. The glare-zone reason string appends `" (Z-adjusted)"` when at least one contributing zone has `z > 0`.

### Default new entry title to source device name (`6ea10ed`)

When a user leaves the name field blank in the create flow and the first selected cover entity has an attached HA device, the entry title now defaults to the device's `name_by_user` or `name` verbatim ŌĆö no "Vertical/Horizontal/Tilt" prefix, no "Adaptive" word. A Zigbee device named "Patio Stairs Shade" produces the entry title "Patio Stairs Shade" and a proxy entity `cover.patio_stairs_shade_managed`. Falls back to the existing `Adaptive {entity_name}` behavior when no device is attached, and respects a user-typed name as before. The lookup lives in the new `_get_device_name_for_entity` function in `config_flow.py`.

## ­¤Éø Fixed

### Forecast tick chain ŌĆö five cooperating fixes

`sensor.<cover>_position_forecast` was frozen at its boot-time value in all but the lightest-activity installations. The root causes formed a chain:

- **Tasks destroyed before first `await` ([#448]):** `hass.async_create_background_task` was called from a sync timer callback, where the task reference is weakly held and can be collected by the GC before the event loop runs it. The symptom was repeated "Task was destroyed but it is pending!" log entries. The fix switches both the initial-compute task and the periodic tick to `config_entry.async_create_background_task`, which pins each task to the entry's lifetime.

- **Tick function dispatched to executor (`f6fe3b0`):** Without `@callback`, HA classifies the plain `def _tick` as `HassJobType.Executor` and runs it in a worker thread. From there, `loop.create_task(..., eager_start=True)` raises `RuntimeError: loop is not the running loop`, so the task is never created and the recompute silently does nothing. Adding `@callback` to `_tick` keeps it on the event loop where it belongs. This was the root cause of the freeze on live instances where the task-pinning fix alone did not restore updates.

- **Ticks staggered per-entry (`3e217e0`):** The scheduler used `async_track_time_interval`, which fires each entry's tick at a fixed offset from its own startup time. Switching to `async_track_time_change` with `minute=range(0, 60, FORECAST_RECOMPUTE_INTERVAL_MIN), second=0` aligns every entry to the wall-clock 5-minute grid (`:00`, `:05`, `:10`, ŌĆ”), so the dashboard sees a single synchronised refresh.

- **Listeners not notified after recompute (`c41c27e`):** `async_recompute_forecast` updated `coordinator.data.position_forecast` via `dataclasses.replace` but never called `async_update_listeners()`. The sensor only got a fresh value on the next regular coordinator cycle ŌĆö up to 4ŌĆō20 minutes later depending on activity. The fix adds `self.async_update_listeners()` immediately after the data field is updated, gated on `self.data is not None` to skip the pre-first-refresh window.

- **FOV-enter/exit events lag the position curve (`03e5dab`):** The event marker for `fov_enter` / `fov_exit` was placed at the first 15-minute forecast sample whose handler had already flipped, which could lag the actual FOV crossing by up to a full step. The new `_refine_fov_crossing` function in `forecast.py` scans SunData's native 5-minute grid between the two bracketing samples and places the event at the first grid point where `direct_sun_valid` actually flips, reducing timing error from ┬▒15 min to ┬▒5 min.

### My Position button bypasses auto-control gate and routes hardware My correctly ([#430], PR [#431])

`AdaptiveCoverMyPositionButton.async_press` was calling `coordinator.async_apply_user_position` without `bypass_auto_control=True`. When the automatic-control switch was off, the `auto_control_off` gate in `_cmd_svc.apply_position` silently dropped the command. The fix passes `bypass_auto_control=True` and `use_my_position=True` from the button to `async_apply_user_position`, which threads them into `_build_position_context`. `use_my_position` is OR-composed with the pipeline result's value so either source can enable the hardware-My routing path.

### Custom-position attrs propagate through pipeline registry ([#421], PR [#424])

`PipelineRegistry.evaluate` was building the `PipelineResult` for the winning handler by copying individual known fields, leaving the two custom-position diagnostic fields (`custom_position_active_slot`, `custom_position_minimum_mode`) stranded in the handler's local result and never reaching `sensor.py`. Switching to `dataclasses.replace` fixes propagation for all current and future fields at once.

### Reconciliation pass honors in-transit guard ([#418], PR [#423])

The `handle_state_change` path in `AdaptiveCoverManager` has rejected mid-travel position events since issue #271, preventing false manual-override detection when HA's reported position lags physical position during a transit move. The reconciliation pass in `CoverCommandService` lacked an equivalent check: a reconcile tick landing while a cover was mid-travel could re-send `set_cover_position` toward the prior ACP target and override a user's RF command. The fix extracts `_is_cover_in_transit` on `CoverCommandService` (reading `hass.states.get(entity_id).state in ("opening", "closing")`) as a shared predicate. The reconciliation loop skips any in-transit entity and records a `reconcile_skipped_in_transit` event.

### `cover_type` now present on `control_status` sensor attributes

`sensor.<cover>_control_status` was missing the `cover_type` attribute. The companion Lovelace card reads this attribute to determine wedge-fill polarity (awnings: extended = full, retracted = empty; blinds: closed = full, open = empty). Without it, every cover fell back to `cover_blind` behavior, so a fully-extended awning rendered as an empty wedge. The fix adds `"cover_type": s._cover_type` to `_control_status_attrs` in `sensor.py`.

## ŌÜÖ’ĖÅ Changed / Performance

- **O(1) forecast index lookup ([#442], PR [#447]):** `_nearest_index` in `forecast.py` replaces a linear scan over ~289 items with a single arithmetic expression: `max(0, min(len(times) - 1, round(delta / step_seconds)))`. The step is parameterised via `SUN_DATA_STEP_SECONDS = 300` in `const.py` so the function stays correct if the cadence changes. The previous linear scan also used a separate `target_tz` variable rather than coercing `target` in-place; the rewrite fixes both issues together.

- **Shared SunData day cache ([#441], PR [#445]):** `sun.py` gains a module-level `_DAY_CACHE: dict[tuple, _SunDayData]` keyed on `(timezone, lat, lon, elevation, date.today())`. All `SunData` instances at the same location share one `pd.date_range` + astral fill per day; a `threading.Lock` guards concurrent fills via double-checked locking. The instance fast path (existing `_cache_day` check) is unchanged as tier 1; the module dict lookup is tier 2 (no lock needed under CPython); under-lock fill is tier 3. Stale (non-today) entries are purged when a new day's entry is stored.

- **Recorder exclusion for heavy sensor attributes (PR [#446]):** `_SensorSpec` gains an `unrecorded_attributes: frozenset[str]` field. A resolver in `sensor.py` composes per-spec subclasses so HA reads `_unrecorded_attributes` at class init. Excluded keys: `actual_positions`, `actual_distances`, `position_explanation` (Cover Position sensor); `manual_covers` (control_status); `trace`, `custom_position_slots`, `enabled_handlers` (decision_trace); `forecast`, `events` (position_forecast); `per_entity` (position_verification). Live state, templates, automation triggers, and the Lovelace card are unaffected ŌĆö only the recorder DB stops persisting these keys.

- **Enums consolidated into `const.py` (`b9c2e53`):** `enums.py` is deleted. `CoverType`, `TiltMode`, `TemperatureSource`, `PresenceDomain`, `ClimateStrategy`, and `ControlMethod` move into a new "Enumerations" section of `const.py`. `SensorType` is replaced by the proper `CoverType(StrEnum)` and all importers are updated. Forecast constants (`FORECAST_STEP_MINUTES`, `FORECAST_WINDOW_HOURS`, `EVENT_SUNRISE`, `EVENT_SUNSET`, `EVENT_FOV_ENTER`, `EVENT_FOV_EXIT`) move from `forecast.py` into `const.py`. No behavior change.
