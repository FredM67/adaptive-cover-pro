# v2.24.0

Fixes a long-standing false manual-override defect on venetians caused by slow-bus publish lag on the position axis, and improves the options flow layout so master toggles surface first in every configuration step.

---

## 🚀 Features

### Options flow fields now follow a consistent master-toggle-first ordering ([#482])

Every step in the options flow now applies a uniform "master toggle → sensors → thresholds → actions" pattern, addressing user feedback that global toggles were buried below the fields they controlled.

`POSITION_SCHEMA` pairs each enable toggle with its value, groups sunset sensors above sunset thresholds, and relocates `CONF_RETURN_SUNSET` — which is semantically a sunset action — from `AUTOMATION_SCHEMA` where it previously lived. `AUTOMATION_SCHEMA` lifts entity sensors above plain-time counterparts now that `CONF_RETURN_SUNSET` has moved. `weather_override_schema` groups all sensors first, then all thresholds, then the action field. `light_cloud_schema` groups sensors above thresholds. `temperature_climate_schema` groups sensors above thresholds. No option keys were renamed; no migrations are needed.

---

## 🐛 Fixes

### Venetian primary-axis publish-lag no longer fires a false manual override ([#33], [#486])

Slow-bus actuators — Somfy IO via Tahoma, sluggish KNX devices, Fibaro or Shelly bridges that republish — can take 30–60 seconds after a cover physically stops to emit a final `current_position` state. The tilt axis already had a publish-lag suppression guard (introduced in PR #408) that recognised this window and blocked the late state from being read as a user touch. The position axis did not, so whenever a tilt command was running, the late `current_position` publish landed outside the tilt guard's scope and was read as a 100% user-initiated move — falsely triggering manual override on venetians.

The fix adds `VenetianPolicy.primary_axis_suppression`, which applies the same sequencer predicate to the position axis that the tilt axis already used. `AdaptiveCoverManager.handle_state_change` now calls this check before the legacy threshold comparison; when it fires, the event is recorded as `manual_override_rejected_primary_axis_suppression` and the suppression is counted against the entity via `AdaptiveCoverManager._record_primary_axis_suppression`. A new per-instance config option, `CONF_VENETIAN_BACKROTATE_PUBLISH_LAG` (default `DEFAULT_VENETIAN_BACKROTATE_PUBLISH_LAG_SECONDS = 45.0`, range `MIN_VENETIAN_BACKROTATE_PUBLISH_LAG = 15.0`–`MAX_VENETIAN_BACKROTATE_PUBLISH_LAG = 180.0` seconds), lets users widen the suppression window for particularly sluggish buses without touching the integration defaults. Per-entity 24-hour suppression totals are surfaced in diagnostics under `primary_axis_suppression_last_24h` via `AdaptiveCoverManager.primary_axis_suppression_counts`, so users can confirm whether the default window is wide enough or needs adjustment.

### Sun-tracking minimum position now enforces when stored as a float ([#475], [#476])

HA's `NumberSelector` can persist numeric option values as floats (`25.0` rather than `25`). The previous guard in `PositionConverter` used `isinstance(sun_tracking_min_pos, int)`, which rejected float values outright, causing the sun-tracking minimum-position floor to silently fall back to `min_pos` whenever the stored value was a float. Users who set `CONF_MIN_POSITION_SUN_TRACKING` to any non-zero value via the UI would see the floor ignored on the next restart.

The guard is now `isinstance(sun_tracking_min_pos, (int, float))`, accepting both types, and the value is `int()`-coerced before use. `CoverConfig.from_options` also coerces `min_pos_sun_tracking` to `int` at load time — using an explicit `is not None` check so that a stored `0` is preserved rather than treated as falsy — so the value arrives at `PositionConverter` as an integer regardless of what `NumberSelector` wrote to storage.

### FOV of 0° on one side is now preserved through the config loader ([#484], [#488])

`CoverConfig.from_options` previously loaded `CONF_FOV_LEFT` and `CONF_FOV_RIGHT` with `options.get(CONF_FOV_LEFT) or 90`. The `or 90` fallback treats any falsy value — including a legitimate `0` — as absent and silently replaces it with 90°. A window that only catches morning sun can reasonably have `fov_left=0`, meaning the sun is never valid on the left side of the azimuth; with the old loader, that setting was discarded on every restart and the integration continued treating the sun as in-view for negative-gamma angles.

The fix switches both fields to an explicit `is not None` guard, preserving `0` as a valid value, and introduces named constants `DEFAULT_FOV_LEFT = 90` and `DEFAULT_FOV_RIGHT = 90` so the fallback value is defined in one place.
