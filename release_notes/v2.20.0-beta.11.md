## ЁЯкЯ Dual-Axis Venetian Blind Support тАФ Beta 11 (closes #33)

Beta.11 fixes custom-position sensor release getting stuck at the slot's
position, tightens manual-override detection for assumed-state and slow
Zigbee covers, adds a `is_sunny` binary sensor option for cloud
suppression, and clarifies the Target Position sensor display during
manual override. All beta.1 through beta.10 content is included.

### тЬи New in Beta 11

- **Custom-position sensor release force-return (#365):** When a
  custom-position sensor turns off, covers now return immediately to
  the calculated position. Previously the time-delta gate blocked the
  move and the cover stayed stuck at the slot's position until the gate
  elapsed. The fix tracks per-sensor on/off state across cycles and
  injects `force=True` with reason `custom_position_released` when the
  released sensor was the active trigger, mirroring the existing
  force-override release pattern. The workaround of toggling Automatic
  Control OFF then ON is no longer needed.

- **`is_sunny` binary sensor for cloud suppression (#363):** An optional
  binary sensor, input boolean, switch, or schedule can now drive the
  `is_sunny` flag consumed by `CloudSuppressionHandler` directly.
  Useful when your weather integration doesn't expose a `condition`
  attribute or when you want a smarter, custom-logic sunny signal.
  When the sensor is unavailable the integration falls back to the
  existing weather-entity logic; installs that don't configure the
  sensor see no behaviour change.

- **Cloud suppression master toggle moved to top of config step (#364):**
  The cloud suppression on/off toggle now appears before the trigger
  sensors it gates in the Light & Cloud configuration step. The toggle
  label and help text are reworded so it is obvious that the field
  enables or disables cloud suppression as a whole. Previously the
  toggle was buried below the sensor fields, which led users to configure
  sensors before realising the feature was off.

- **Target Position shows held position during manual override (#33):**
  `ManualOverrideHandler` now populates `held_position` on the pipeline
  result from the cover's actual current position. The Target Position
  sensor and diagnostics download show where the cover physically sits
  during override rather than the solar-tracking value being shadowed.
  A cover held at 100% by the user no longer shows a target of 20%.

### ЁЯРЫ Bug Fixes

- **Custom-position sensor release stuck at slot position (beta.11,
  #365):** `_handle_custom_position_release` in `coordinator.py` now
  sets `force=True` on the outbound position command whenever the
  released sensor was the trigger entity in the prior cycle. The
  time-delta gate is bypassed only on the release cycle; subsequent
  normal-tracking cycles resume gating as usual.

- **False manual override on assumed-state covers (beta.11):**
  `PositionContextTracker` mints an explicit `Context` for every
  `set_cover_position`, `open_cover`, and `close_cover` call and records
  its id. State-change events whose context id is absent from the tracker
  and whose `user_id` is non-null are marked manual immediately, before
  the position-math path can be defeated by post-reconcile state. Covers
  reported by Bond/Olibra and OPEN/CLOSE-only entities (no position
  feedback) are now handled correctly.

- **False manual override on slow Zigbee covers (beta.11, #361):**
  `handle_state_change` records `manual_override_rejected_in_transit`
  and returns early when the cover's own state attribute is
  `opening` or `closing` at event time. The settling event (with the
  true final position) still runs the full check, so real manual moves
  are caught one event later. Tilt-axis changes during a position
  traverse are unaffected.
