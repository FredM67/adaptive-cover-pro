# v2.23.1

Two small but sharp fixes for users on edge-case configurations: one silently corrupted a user-set distance of 0 m, the other let floor-mode rules slip through inconsistently depending on which pipeline handler won.

---

## 🐛 Bug Fixes

### Distance = 0 silently replaced with 1.0 m ([#464], [#466])

If you set your shaded distance to 0 m, the integration ignored it and used the default 1.0 m instead. The root cause was an `or` fallback in `configuration_service.py` — `options.get(CONF_DISTANCE) or 1.0` treats `0` as falsy, so any zero value was silently overwritten. The same pattern affected `CONF_HEIGHT_WIN`. The fix introduces `DEFAULT_DISTANCE` in `const.py` and rewrites both reads to use `_raw_distance if _raw_distance is not None else DEFAULT_DISTANCE`, so a user-set 0 is preserved exactly as stored.

### Floor-mode rules applied inconsistently across handlers ([#463], [#465])

Covers using floor-mode (minimum-position) rules could behave unpredictably depending on which pipeline handler won — custom-position, force-override, and weather handlers each carried their own ad-hoc floor checks, and the results didn't always compose correctly. The fix extracts floor enforcement into a dedicated post-pipeline clamp: after the override-priority chain picks a winner, `apply_minimum_mode()` in the new `pipeline/floors.py` module enforces all active floors on top via `FloorClampInfo` and `gather_active_floors()`. The decision trace is clearer and the duplicate logic across handlers is gone.
